description: "Deep dive on every artifact kind generated by Promptster, including field definitions and how to use them for candidate evaluation."

When a candidate completes a session, Promptster's worker pipeline generates several artifact kinds. Each captures a different dimension of the candidate's work. This guide explains what each artifact contains and how to interpret it.

## analytics_v1

Raw behavioral metrics computed from the event stream. Contains 40+ quantitative data points.

### Key fields

| Field | Type | Description |
|-------|------|-------------|
| `promptCount` | integer | Total number of prompts sent to the AI tool |
| `sessionDurationMs` | integer | Wall-clock time from first to last event |
| `commandFailRate` | float | Fraction of terminal commands that exited non-zero |
| `uniqueFilesTouched` | integer | Distinct files modified during the session |
| `verifyIntensity` | float | Ratio of verification actions (test runs, manual checks) to total actions |
| `manualEditRatio` | float | Fraction of file changes made by hand vs. AI-generated |
| `firstChangeLatencyMs` | integer | Time from session start to first file modification |
| `avgPromptLength` | float | Average character count of prompts |
| `maxPromptLength` | integer | Longest single prompt |
| `commandCount` | integer | Total terminal commands executed |
| `testRunCount` | integer | Number of test suite executions |
| `aiSuggestAcceptRate` | float | Rate of AI suggestions accepted without modification |
| `humanTerminalCommands` | integer | Commands typed directly in the terminal (not via AI) |
| `distinctToolsUsed` | string[] | List of AI tools invoked (e.g., `["Read", "Write", "Bash"]`) |

<Note>
All rate/ratio fields are between 0.0 and 1.0. Duration fields are in milliseconds.
</Note>

## data_points_v1

Aggregated work-style analysis with behavioral descriptors. Built on top of `analytics_v1`.

### Structure

```json
{
  "points": {
    "sessionDurationMs": 4500000,
    "promptCount": 47,
    "commandFailRate": 0.12,
    "verifyIntensity": 0.35,
    "manualEditRatio": 0.08,
    "firstChangeLatencyMs": 180000,
    "testResults": {
      "passed": 12,
      "failed": 0,
      "skipped": 2,
      "total": 14
    }
  },
  "processDescriptors": [
    "test-driven",
    "AI-delegating",
    "iterative"
  ]
}
```

### Process descriptors

The `processDescriptors` array contains tags that characterize the candidate's working style:

| Descriptor | Meaning |
|-----------|---------|
| `test-driven` | Ran tests frequently and early in the session |
| `AI-delegating` | Relied heavily on AI-generated code with minimal manual edits |
| `iterative` | Made many small changes with frequent verification |
| `manual-coder` | High ratio of hand-written edits vs. AI suggestions |
| `planner` | Spent significant time reading and exploring before first change |
| `debugger` | High frequency of error recovery and fix-retry cycles |

## prompt_analysis_v1

LLM-powered analysis of the candidate's prompting strategy across five dimensions. Each dimension is rated with a tier and supporting evidence.

### Dimensions

| Dimension | What it measures |
|-----------|-----------------|
| `decomposition` | How well the candidate breaks down the problem into manageable subtasks |
| `contextGathering` | Whether the candidate reads relevant files, docs, and tests before making changes |
| `courseCorrection` | How the candidate responds to errors, test failures, and unexpected results |
| `constraintSpecification` | Whether prompts include specific technical requirements, edge cases, and acceptance criteria |
| `codebaseGrounding` | Whether the candidate references existing patterns, types, and conventions in the codebase |

### Tiers

Each dimension receives one of four tiers:

| Tier | Meaning |
|------|---------|
| `strong` | Consistently demonstrated throughout the session with clear evidence |
| `adequate` | Present in the session but inconsistent or could be more thorough |
| `developing` | Minimal evidence, significant room for improvement |
| `insufficient_evidence` | Not enough data to assess (session too short or limited prompts) |

### Example

```json
{
  "strategies": {
    "decomposition": {
      "tier": "strong",
      "evidence": "Broke the rate limiter into four distinct subtasks: Redis client setup, sliding window algorithm, middleware wiring, and test coverage.",
      "promptExamples": ["First, let me understand the existing middleware chain..."]
    },
    "contextGathering": {
      "tier": "adequate",
      "evidence": "Read the main router file and config but missed the existing rate limit tests.",
      "promptExamples": ["Read the middleware directory and show me how auth middleware is structured"]
    }
  }
}
```

## tool_proficiency_v1

Analysis of how effectively the candidate uses AI coding tools, measured across seven proficiency dimensions.

### Dimensions

| Dimension | What it measures |
|-----------|-----------------|
| `contextManagement` | How well the candidate provides and manages context for AI tools |
| `toolSelection` | Whether the right tool is used for each task (Read vs. Grep vs. Bash, etc.) |
| `subagentUsage` | Effective use of sub-agents and task delegation |
| `planningDiscipline` | Use of TodoWrite, structured planning, and task tracking |
| `promptCraft` | Quality, specificity, and clarity of prompts |
| `errorRecovery` | How the candidate handles tool failures and unexpected outputs |
| `iterationStrategy` | Whether the candidate iterates effectively (small steps vs. large rewrites) |

### Scoring

Each dimension receives a numeric score from 1-5 with descriptive reasoning:

```json
{
  "dimensions": {
    "contextManagement": {
      "score": 4,
      "reasoning": "Consistently read relevant files before making changes. Used Grep to find related code patterns."
    },
    "toolSelection": {
      "score": 3,
      "reasoning": "Generally appropriate tool choices, but used Read for large files where Grep would have been more efficient."
    }
  },
  "overallScore": 3.7,
  "summary": "Effective AI tool user with strong context management and room to improve tool selection."
}
```

## key_moments_v1

Identified turning points in the session -- moments where the candidate's approach shifted, a breakthrough occurred, or an important decision was made.

### Structure

```json
{
  "moments": [
    {
      "timestamp": "2026-04-07T12:15:00.000Z",
      "offsetMs": 900000,
      "kind": "breakthrough",
      "summary": "Identified the root cause: the scheduler uses local time for DST calculations instead of UTC",
      "relatedEvents": ["event-uuid-1", "event-uuid-2"]
    },
    {
      "timestamp": "2026-04-07T12:30:00.000Z",
      "offsetMs": 1800000,
      "kind": "pivot",
      "summary": "Abandoned moment.js approach in favor of Temporal API after discovering compatibility issues"
    }
  ]
}
```

### Moment kinds

| Kind | Description |
|------|-------------|
| `breakthrough` | Candidate identifies root cause or key insight |
| `pivot` | Significant change in approach or strategy |
| `struggle` | Extended period of difficulty or repeated failures |
| `verification` | Candidate validates their solution (running tests, manual check) |

## test_results_v1

Results from the automated test suite run during submission.

```json
{
  "passed": 12,
  "failed": 1,
  "skipped": 2,
  "total": 15,
  "duration_ms": 4500,
  "details": [
    {
      "name": "should rate limit after threshold",
      "status": "passed",
      "duration_ms": 120
    },
    {
      "name": "should handle concurrent requests",
      "status": "failed",
      "error": "Expected 429, received 200",
      "duration_ms": 350
    }
  ]
}
```

<Tip>
For assessments using the OSS issue library, test results directly indicate whether the candidate fixed the bug. All tests passing means the fix is correct.
</Tip>

## candidate_summary

A concise, human-readable summary of the candidate's session generated by Claude Sonnet.

```json
{
  "headline": "Methodical debugger who systematically isolated the DST issue and delivered a clean fix with comprehensive tests",
  "body": "Alice approached the timezone bug by first reading the scheduler's time handling code and identifying three potential failure points. She narrowed down the root cause within 20 minutes by writing targeted test cases for each hypothesis. Her fix replaced local time arithmetic with UTC-based calculations using the Temporal API, and she added 6 new test cases covering edge cases including spring-forward, fall-back, and timezone boundary transitions. Her prompting style was precise and well-scoped, consistently providing file context and specific requirements."
}
```

## interview_followups_v1

AI-generated follow-up questions tailored to the candidate's session, suitable for a technical debrief interview.

```json
{
  "questions": [
    {
      "question": "You chose the Temporal API over moment.js for timezone handling. What trade-offs did you consider, and how would you handle browser support?",
      "context": "Candidate pivoted from moment.js to Temporal API at minute 30",
      "focus_area": "technical_decisions",
      "difficulty": "medium"
    },
    {
      "question": "Your test suite covers DST transitions but not leap seconds. How would you approach testing calendar edge cases at scale?",
      "context": "Candidate wrote 6 test cases focused on DST",
      "focus_area": "testing_strategy",
      "difficulty": "hard"
    }
  ]
}
```

### Fields

| Field | Description |
|-------|-------------|
| `question` | The follow-up question text |
| `context` | What the candidate did that prompted this question |
| `focus_area` | Category: `technical_decisions`, `testing_strategy`, `architecture`, `debugging`, `communication` |
| `difficulty` | `easy`, `medium`, or `hard` |

## Checking artifact availability

Not all artifacts may be present for every session. Check the `kind` field to determine what is available:

```python
artifacts = fetch_artifacts(session_id)
available = {a["kind"] for a in artifacts}

if "prompt_analysis_v1" in available:
    analysis = next(a for a in artifacts if a["kind"] == "prompt_analysis_v1")
    # Process prompt analysis
else:
    # Session may not have enough data, or pipeline is still running
    pass
```

<Warning>
Sessions with fewer than 3 prompts and 0 file diffs will produce limited artifacts. The `analytics_v1` artifact may still be generated, but LLM-powered artifacts like `prompt_analysis_v1` and `candidate_summary` require sufficient data.
</Warning>
