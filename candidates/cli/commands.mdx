Run `promptster` with no arguments to open an interactive menu, or pass a command directly:

```bash
promptster <command> [flags]
```

The commands you'll use in a normal assessment are **`start`**, **`brief`**, **`explain`**, and **`done`**. The rest are for checking status or recovering from a problem.

## Core commands

<AccordionGroup>
  <Accordion title="promptster start <key> â€” begin your assessment">
    Redeems your key, sets up your workspace, configures your editor, and starts the session. This is the first command you run.

    ```bash
    promptster start PST-ABCD-1234
    ```

    **What `start` does:**

    <Steps>
      <Step title="Redeems your key & accepts terms">Validates the key, then walks you through the data-capture notice and the optional authorship-timing question.</Step>
      <Step title="Prepares your workspace">Uses your current folder, a path you choose, or a fresh `~/promptster-test`. If the assessment includes a starter repo, it's cloned at the correct commit.</Step>
      <Step title="Writes TASK.md">Saves the task brief to `TASK.md` and adds `.promptster/` to `.gitignore`.</Step>
      <Step title="Configures hooks & proxy">Writes Claude Code / Cursor hooks into your workspace and routes the AI through the Promptster proxy (the hiring team covers the cost).</Step>
      <Step title="Smoke-tests the proxy">Fires one tiny request so any auth/network problem surfaces now, not on your first prompt.</Step>
    </Steps>

    **Flags:**

    | Flag | Description |
    |------|-------------|
    | `--accept-tos` | Accept the terms non-interactively (scripted/CI use). |
    | `--workspace PATH` | Use `PATH` as the workspace instead of prompting. |
    | `--restart` | Offer to restart a running editor so it reloads the new hooks. |
    | `--verbose` | Print each setup sub-step (paths, proxy URL, hook events) for debugging. |

    ```bash
    promptster start PST-ABCD-1234 --accept-tos --workspace ~/my-assessment
    ```

    <Warning>
      After `start`, open Claude Code or Cursor **from the workspace directory in a new window**. Hooks are scoped to that folder â€” an editor that was already open won't capture your session. Use `--restart` (or restart your editor yourself) if it was already running.
    </Warning>
  </Accordion>

  <Accordion title="promptster explain â€” record a decision">
    Captures the *why* behind a choice. Three ways to use it:

    ```bash
    promptster explain                              # opens an interactive editor
    promptster explain "chose X over Y because..."  # inline, one-liner
    promptster explain --last 30m                   # widen the recent-activity window
    ```

    Inside Claude Code you can also use the `/explain` slash command, which calls this under the hood. The CLI shows a summary of your recent activity for context, then records your note as part of your session.

    | Flag | Default | Description |
    |------|---------|-------------|
    | `--last <duration>` | `20m` | How far back to summarize recent activity (e.g. `15m`, `1h`). Clamped to 1â€“60 min. |
    | `--quiet` | off | Suppress the activity box and styling. Used by the `/explain` slash command; you can use it for clean scripted output. |

    See [Decisions](/candidates/cli/decisions) for the full workflow.
  </Accordion>

  <Accordion title="promptster brief â€” see your task (alias: task)">
    Shows the task brief, time limit, time remaining, and workspace path.

    ```bash
    promptster brief
    promptster brief --json   # machine-readable
    ```

    The same brief lives in `TASK.md` in your workspace. The assessment title is deliberately omitted so it can't leak hints about the solution.
  </Accordion>

  <Accordion title="promptster done â€” submit (flag: --auto)">
    Bundles and uploads your workspace, flushes remaining events, and marks the session complete.

    ```bash
    promptster done
    promptster done --auto   # skip the pending-decision check
    ```

    On success you get a results link (copied to your clipboard), and your hooks, shell hook, and proxy config are cleaned up automatically. The workspace is then safe to delete.

    <Warning>
      If you started decision notes that are still empty, `done` refuses to proceed until you finish them with `promptster explain` (or pass `--auto`). If the upload fails, `done` exits with an error rather than falsely reporting success â€” re-run it once your connection is back.
    </Warning>
  </Accordion>
</AccordionGroup>

## Status & diagnostics

<AccordionGroup>
  <Accordion title="promptster status â€” session info">
    Session ID, start time, elapsed time, time remaining (if a limit is set), and the live event count the server has received.

    ```bash
    promptster status
    promptster status --json
    ```

    Use the event count to confirm capture is working â€” it should climb as you prompt and edit. If it's stuck at zero, see [Troubleshooting](/candidates/troubleshooting#my-events-arent-being-captured).
  </Accordion>

  <Accordion title="promptster doctor â€” diagnose your setup">
    Runs a full health check and prints each result with `âś“` (pass) or `âś—` (fail) plus a suggested fix. It checks:

    - `git` and the `claude` binary are on your `PATH`
    - the `promptster` binary is installed and `~/.promptster/bin` is on your `PATH`
    - no conflicting `ANTHROPIC_API_KEY` / `ANTHROPIC_AUTH_TOKEN` is exported in your shell (these break the proxy)
    - your session is active and not expired, and the `apiKeyHelper` can resolve a token
    - Claude Code hooks and the proxy config are present in your workspace
    - the shell hook is installed and sourced
    - the API is reachable and your session is fetchable

    ```bash
    promptster doctor
    ```

    Run this first whenever something seems off.
  </Accordion>

  <Accordion title="promptster verify â€” check your session is tamper-evident">
    Confirms that the events recorded for your session are cryptographically signed and form an unbroken chain.

    ```bash
    promptster verify
    ```

    Output is either `âś“ N / N events signed Â· chain intact` or a clear failure. Sessions started with an older CLI may be unsigned, which the command reports plainly.
  </Accordion>

  <Accordion title="promptster version">
    Prints the installed CLI version.

    ```bash
    promptster version
    ```
  </Accordion>
</AccordionGroup>

## Recovery

<AccordionGroup>
  <Accordion title="promptster abort â€” discard without submitting (alias: cleanup)">
    Tears down the current session, hooks, shell hook, and proxy config **without** submitting any code or completing the assessment.

    ```bash
    promptster abort
    ```

    Use this if you want to walk away from a session cleanly. It does not flip your key's status, so reach out to the hiring team if you intended to submit.
  </Accordion>

  <Accordion title="promptster reset [--purge] â€” fix a broken setup">
    Like `abort`, but also sweeps leftover global config under `~/.promptster`. It resolves your workspace even when the session file is corrupt, so it works when nothing else does.

    ```bash
    promptster reset          # keep the installed binary
    promptster reset --purge  # also remove the binary (full uninstall)
    ```

    See [Recovering from a broken setup](/candidates/troubleshooting#recovering-from-a-broken-setup).
  </Accordion>

  <Accordion title="promptster env --clear â€” clear stale Anthropic env vars">
    Emits shell `unset` lines for `ANTHROPIC_*` variables left behind by an older session. Run it if `doctor` reports a conflicting export:

    ```bash
    eval "$(promptster env --clear)"
    ```

    (Bare `promptster env` is an internal session-expiry check â€” it prints nothing.)
  </Accordion>
</AccordionGroup>

## Interactive menu

Running `promptster` with no arguments opens a numbered menu. Type a number, or paste your `PST-XXXX-XXXX` key directly â€” the CLI detects the prefix and routes to `start`.

```

   5  Submit assessment      Upload code and complete the session
   6  Run doctor             Diagnose your local setup
   7  Abort assessment       Discard the session without submitting
   8  Help                   Show all available commands
```
